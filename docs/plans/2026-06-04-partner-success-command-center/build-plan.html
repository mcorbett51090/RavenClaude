Spending the fewest credits

Codex bills roughly per turn and per token. Five habits cut the bill hard.

1 · One prompt = one deliverable Each prompt is self-contained — full file list, schema, and acceptance criteria. Paste it, let Codex produce everything in one response. Don't trickle requirements in.
2 · Don't let it explore Every prompt ends with a Constraints line: don't ask questions, don't run the app, don't install anything, produce the files directly. Exploration is where credits evaporate.
3 · Stop when it's enough Prompts 1 → 2 give you a dashboard you use every morning. Prompts 3–6 are bonus depth. You don't need all six.
4 · Paste the schema, don't re-derive it Prompt 1 prints the data contract. Later prompts say "read field names from the data.json already in the repo" so Codex reads one file instead of reasoning it out again.
5 · Synthetic data first The dashboard is fully usable on fake data. Wiring real systems is a separate, much larger effort — do it only once the layout is proven.

One gotcha — already solved in the reference build

Browsers block fetch() of a local file under file://, so a dashboard that fetches data.json won't load by double-clicking. Serve the folder (python3 -m http.server 8000 → http://localhost:8000/dashboard.html), or ask Codex to inline the data as a data.js window.__DATA__ script. Prompt 2 tells Codex to show that exact tip in its error state.

Three load-bearing decisions

The choices that shape everything else — already made, so Codex doesn't have to.

S1 · It's a static site

Plain HTML + a data.json + a Python generator. No framework, no bundler, no server. Opens in a browser. This is the fastest, cheapest thing to build and the easiest to hand to someone else.

S2 · One identity spine

Every partner has one account_uid; every record (contacts, tickets, usage, contracts) joins to it. When real data lands, all five systems map onto this one key. For the synthetic build it's just a clean primary key.

S3 · A tunable priority score

A 9-signal weighted score (0–100) decides who to call first. The weights live in priority_weights in the data file so you can re-tune without touching code — and every row shows which signal drove the score, so it's never a black box.

1  Data foundation Essential

The contract every later prompt reads against. Prints the schema once so nothing downstream re-derives it.

Prompt 1 · Schema + synthetic datamedium 
Build the data foundation for a single-tenant K-12 edtech Partner Success dashboard. Produce TWO files in one response: data/synthesize.py and a freshly generated data/data.json.

data/synthesize.py generates a realistic, fully SYNTHETIC, FERPA-safe sample of 25 partner districts. Requirements:
  - Reproducible: seed the RNG with 42 so output is byte-identical across runs.
  - No real names. Use generated district names like "Maple Grove USD", "Riverbend School District".
  - Only aggregate counts — NEVER any student-level or personal data.
  - Spread the sample so it exercises the dashboard:
      * >=3 partners in each health band: green (>=70), yellow (50-69), red (<50)
      * >=2 partners in each renewal bucket: 180 / 120 / 90 / 60 / 30 days to renewal
      * a mix of segments and a handful flagged top15_status = true
  - Write the result to data/data.json (pretty-printed, 2-space indent).

The JSON shape (this is the contract — emit EXACTLY these blocks and field names):

priority_score math: priority_score = sum(priority_breakdown[k] * priority_weights[k]) / 100, rounded to an integer, clamped 0-100. priority_breakdown[k] is the raw 0-100 strength of each signal (e.g. a 30-day renewal scores ~95 on renewal_timing). Keep it deterministic.

Generate 2-6 contacts, 4-10 timeline_events, ~30 days of usage_daily, 1-3 success_plans, 1 contract, 0-4 tickets, and 2-5 calendar_events per partner.

2  Daily Operating System Essential

The dashboard you open every morning. Answers "who do I call today and why" in under 15 seconds.

Prompt 2 · dashboard.html (4 widgets)large 
Build dashboard.html — a self-contained, single-page "Partner Success Command Center" that fetches ./data/data.json and renders the daily operating system for a K-12 edtech PSM. Plain HTML + CSS + vanilla JS in ONE file. No framework, no CDN except the system font stack. Load the data with fetch on ./data/data.json. IMPORTANT: browsers block fetch of a local file under file://, so if the fetch fails, render a friendly message telling the user to run `python3 -m http.server 8000` from this folder and open http://localhost:8000/dashboard.html (do NOT silently show a blank page).

Visual style: calm, professional, Intercom-like. Cool near-white canvas (#f5f6f8), white cards with a 1px hairline border and very soft shadow, teal accent (#1f7f78), Inter/system sans. Health colors: green >=70 (#1f7a3f), yellow 50-69 (#9a7010), red <50 (#b03f3f) — read the thresholds from data.bands, never hardcode them.

Render these FOUR widgets, top to bottom, each in its own card:

1. PORTFOLIO SUMMARY — a row of stat tiles, each a count derived from partners[]:
   Total · Active · Top 15 · In Renewal Window (renewal_date within 120 days) · At Risk (health_score < 50) · Open Escalations (sum) · Need Outreach (last_touchpoint_at older than 90 days).

2. PORTFOLIO HEALTH SNAPSHOT — average health_score, average sentiment_score, average engagement_score (big numbers, colored by band), plus "X partners with declining usage" and "X partners with no touchpoint in 90+ days".

3. DAILY ACTION CENTER — the heart of it. A table of ALL partners sorted by priority_score descending. Columns: Partner · Priority (the score as a colored chip) · Why (action_reason plus the TOP contributing signal from priority_breakdown shown as "renewal_timing 38%") · Recommended Action (recommended_action) · Due (next_required_touchpoint_at as a date). The top 5 rows are visually emphasized. Every "Why" cell must name the signal that drove the score — never show a score without its reason.

4. CALENDAR / UPCOMING TOUCHPOINTS — a countdown list derived from each partner's renewal_date and next_required_touchpoint_at: e.g. "Maple Grove USD — 30 days until renewal outreach", "Riverbend SD — 14 days until next check-in". Sort soonest-first, show the next ~15. Add a small note that these countdowns are derived from contract dates and cadence rules, not synced calendar invites.

Also include a compact top bar with the title "Partner Success Command Center", the data.generated_at timestamp, and a one-line legend for the green/yellow/red bands.

Make it responsive (stat tiles wrap on narrow screens) and keep all numbers honest — if a value is missing in the data, show "—", not a guess.

Constraints: Do not ask clarifying questions. Do not run a server or install anything. Read field names from the data/data.json already in the repo. Produce the single dashboard.html file in one response, then stop.
You get: a working dashboard.html. Double-click it. With data/data.json present, the four widgets light up. This is the minimum viable deliverable — you can stop here.

After the build — wiring real data

The dashboard renders on data/data.json. Going from synthetic to real data means replacing that one file — dashboard.html never changes.

Six questions to answer before real-data work

Which support tool (Zendesk/Freshdesk/Intercom/SFDC Service Cloud)? Which contract system (Salesforce CPQ/Ironclad/DocuSign)? Which calendar (Google/Outlook)? Where does the Top 15 list live today? Is scoping just you or your team? Is sentiment Planhat-native or PSM-set? Five minutes of answers decides the connector scope.

The daily loop, two ways:

  • Synthetic (now): python3 data/synthesize.py → open dashboard.html.
  • Real (later): export real data → data/data.json → open dashboard.html. Same file, same dashboard.
