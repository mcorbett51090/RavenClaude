# Visual settings UI for non-technical users — plan

_Two-agent planning exercise, 2026-05-26. Two architects independently produced a plan (path + setup + build plan + gap analysis); a third independent agent cross-reviewed both and reconciled them. This document records all three._

## Problem

RavenClaude's `ravenclaude-core` plugin ships a **comfort-posture** permission system and a **command-review tribunal**. These settings affect **Claude Code** only (the agentic coding tool) — not claude.ai chat, not ChatGPT. Today they are edited by hand or via `/set-posture`. We want **non-technical users** (no git/repo knowledge) to tweak them through a **visual interface**, on two surfaces: **Claude Code local** (terminal/desktop/IDE) and **Claude Code on the web** (claude.ai/code).

### Load-bearing facts (verified against the repo + the Claude Code web docs)

- The editor UI **already ships in the plugin**: `plugins/ravenclaude-core/dashboard.html` (~412 KB, self-contained, generated by `scripts/generate-dashboards.py`). On `/plugin install` it lands in the consumer's plugin cache — **UI distribution is solved.**
- The translator **ships too**: `plugins/ravenclaude-core/scripts/apply-comfort-posture.py` (the `/set-posture` command) already takes `--project-root` and has a `find_project_root` walk-up.
- The local companion server does **NOT** ship: `scripts/serve-dashboards.py` lives only at the **marketplace repo root** (excluded from the plugin) and hard-codes `REPO_ROOT` as serve-dir, `/__save` write-target, **and** the `--project-root` it passes the translator (line 154). A consumer can't use it against their own project. **This is the single biggest blocker.**
- The dashboard's "no-server" help (`dashboard.html` ~line 1249) tells the user to run `python3 scripts/serve-dashboards.py` — a script a consumer doesn't have. **Dead-end copy.**
- **Two-artifact wrinkle:** the dashboard writes `.ravenclaude/comfort-posture.yaml`; nothing reaches `.claude/settings.json` until the translator runs. The server's `POST /__save` auto-runs it; the File System Access (FSA) and download save paths do **not** (they need a follow-up `/set-posture`).
- **FSA** (`showSaveFilePicker` + IndexedDB handle) is Chromium-only, secure-context-only, gesture-gated, and writes only to the user's **local** machine.
- **Tribunal config** (`command_review:` / `thing:` toggles) lives in the **same** YAML the dashboard already writes and is read directly by `thing-orchestrator.sh` — so it needs **no new translation path**, just the same save+commit.
- **Web surface:** the container is ephemeral and clones the repo; **only repo-committed** `.claude/settings.json` / hooks / skills / commands / `CLAUDE.md` carry over ("in the cloud, only hooks committed to the repo run"). A per-user GUI edit **cannot persist without a commit**. (Source: https://code.claude.com/docs/en/claude-code-on-the-web.)

## Plan A (architect 1)

**Most desirable path — local:** companion server launched by a slash command (it already auto-applies the translator on save), with a hosted static dashboard + FSA as a no-install secondary for Chromium users, and a conversational/preset flow as the plain-language fallback. **Web:** inherited committed config + a SessionStart auto-apply hook — the only model that persists in an ephemeral container; novices inherit a posture and touch nothing. Self-service GUI editing on web can't persist without a commit, so the in-session agent editing the YAML and committing it is the only self-service bridge.

**Setup:** a slash command runs `serve-dashboards.py` against the consumer project + plugin dashboard, opens the browser; save → `/__save` writes `.ravenclaude/comfort-posture.yaml` and re-runs the translator. Web: a committed SessionStart hook runs the translator when a committed `comfort-posture.yaml` exists; wire via `/init-agent-ready`.

**Build plan:** (1) make `serve-dashboards.py` consumer-aware (`--project-root`, serve dashboard from plugin root, write to consumer root); (2) launcher slash command; (3) dashboard "Simple mode" (presets + plain-language toggles default; per-layer grid + tribunal panel behind Advanced); (4) web SessionStart auto-apply hook + default posture, wired into `/init-agent-ready`; (5) consumer docs; (6) verify via audit-gates (new fixtures), prettier, layout allow-list, version mirror bump.

**Gap analysis:** the dashboard + server are marketplace-dev tools, not a consumer product (server hardcodes the marketplace root); static host can't translate (405 on `/__save`); web self-service can't persist without a commit; the per-layer × per-permission grid + tribunal panel are dev-grade and will stall a novice (needs a Simple-mode default); discoverability + the `python3` dependency; the server has no auth (keep localhost-only).

## Plan B (architect 2)

**Most desirable path — local:** Approach B (companion server launched by a slash command), with a conversational/preset flow as the zero-GUI fallback. localhost is a secure context, so FSA's limits evaporate and all browsers work; `/__save` already auto-applies, collapsing the two-artifact wrinkle to one click. **Web:** Approach E (committed config consumed via a SessionStart hook), explicitly **not** a server — the container can't persist a per-user edit. Crux: **local = interactive GUI that writes+applies; web = inherit the committed artifact.**

**Setup:** a shipped, project-root-aware server `plugins/ravenclaude-core/scripts/serve-posture.py` that **splits** the single `REPO_ROOT` into `SERVE_DIR` (plugin root) and `PROJECT_ROOT` (consumer project, the write-target + translator root); a launcher `commands/posture-dashboard.md`. Web: author locally, commit the YAML + the translated `settings.json`; a committed SessionStart hook re-applies at session start, wired by `/init-agent-ready`.

**Build plan:** (1) fork/ship the project-aware server; (2) launcher command; (3) fix the dead-end no-server copy + add "now run /set-posture" to FSA/download toasts, regenerate `dashboard.html`; (4) web SessionStart re-apply hook + default posture via `/init-agent-ready`; (5) conversational preset branch in `set-posture` skill; (6) version mirror bump; verify via json.tool, `generate-dashboards.py --check`, prettier, audit-gates fixtures, manual test in a separate project.

**Gap analysis:** missing a shippable project-root-aware server (headline blocker); missing launcher command; dead-end degraded-path copy; no web re-apply story. Risks: serve-dir vs project-root path-confusion (move the traversal guard to `PROJECT_ROOT`, re-fuzz); server-in-web-container is a wrong turn; Codespaces `~/.claude` is ephemeral (prefer the project layer); keep `127.0.0.1`. Open: commit YAML only vs YAML+`settings.json` (lean both); `/posture-dashboard` command vs `/set-posture --ui` flag.

## Cross-review & reconciliation (independent reviewer)

### Where they agree (high-confidence)

- **Local = companion HTTP server launched by a slash command, with auto-apply** — localhost is a secure context (FSA limits irrelevant, all browsers work); `/__save` already writes the YAML **and** re-runs the translator, collapsing the two-artifact wrinkle to one click. The job is "finish/ship the server," not "build one."
- **Web = committed config consumed via a SessionStart hook, never a server** — the container can't persist a per-user edit without a commit. The visual editor is a **local authoring tool**; the web **inherits** the committed artifact.
- **Conversational/preset fallback** is the zero-GUI bridge — and the only self-service that can commit on web.

### What each missed (verified)

- `serve-dashboards.py` ships in the plugin? **No** (marketplace-root only; hardcodes `REPO_ROOT` incl. line 154). Plan B precise; **Plan A muddled** — "make it consumer-aware" wrongly implies editing it in place ships it.
- `dashboard.html` ships in the plugin? **Yes.** Plan B headlines it; **Plan A omits** that distribution is already solved.
- Dead-end no-server copy (line ~1249): **Plan B caught, Plan A missed.**
- **Plan A's compensating strength:** the "Simple mode" UX point — the per-layer × per-permission grid + tribunal panel is dev-grade and will stall a novice. **Plan B under-weighted this.**
- **Plan B leans to a fork** (`serve-posture.py`); a fork doubles maintenance — **refactor** the existing script instead (split `SERVE_DIR` vs `PROJECT_ROOT`, ship that).
- **Both missed:** tribunal toggles need **no new translation path** (same YAML); `python3` is a hard dep (launcher must degrade cleanly); on web, committing the YAML is pointless unless `settings.json` is also committed or the hook re-applies.

### Reconciled recommendation

- **Local:** companion server launched by a slash command + conversational/preset fallback. **Refactor (don't fork)** `serve-dashboards.py` to split serve-dir from project-root, ship it under `plugins/ravenclaude-core/scripts/`, add a launcher, and default the dashboard to **presets-first Simple mode** (Advanced hidden).
- **Web:** committed config + SessionStart re-apply hook, never a server. Commit **both** `comfort-posture.yaml` and the translated `settings.json` (belt-and-suspenders); document "configured once locally, inherited by every web session."

### Unified build sequence (supersedes both)

1. **Refactor the server into the plugin** (`plugins/ravenclaude-core/scripts/serve-posture.py`): split `REPO_ROOT` → `SERVE_DIR` (plugin root, serves `dashboard.html`) + `PROJECT_ROOT` (`--project-root` → `$CLAUDE_PROJECT_DIR` → walk-up; write-target + translator root). **Move the traversal guard to `PROJECT_ROOT`**; keep `ALLOWED_TARGETS`, the 5 MB cap, and auto-apply.
2. **Launcher command** `plugins/ravenclaude-core/commands/posture-dashboard.md` (allowed-tools: Bash): runs the server with `--project-root "$CLAUDE_PROJECT_DIR"`, prints the URL, degrades cleanly if `python3` is absent.
3. **Fix dead-end copy** in `scripts/generate-dashboards.py` (no-server-help → "run `/posture-dashboard`"; add "now run `/set-posture`" to FSA/download toasts). Regenerate `dashboard.html`.
4. **Simple mode:** presets + plain-language category toggles as the default; per-layer grid + tribunal panel behind "Advanced." Regenerate; keep the freshness gate green.
5. **Web inheritance:** committed SessionStart re-apply hook template + a sane default `comfort-posture.yaml`, wired by `/init-agent-ready` (`commands/init-agent-ready.md` + `templates/agent-ready-repo/`). Document web=inherit in both `CLAUDE.md` files.
6. **Conversational fallback:** preset branch in `plugins/ravenclaude-core/skills/set-posture/SKILL.md` (Recommended/Cautious/Fast → write YAML + apply).
7. **Version mirror bump** (`plugin.json` + `marketplace.json`; CI enforces).

**Verification gates:** `python3 -m json.tool` on manifests; `bash -n` + exec-bit on new shell; `generate-dashboards.py --check` (freshness); `prettier --write . && prettier --check .` (whole tree); layout allow-list (`plugins/*/scripts/**` + `plugins/*/commands/**` already cover the new files — no glob edit); `scripts/audit-gates.sh` with a **new bidirectional fixture pair** (server writes to the consumer root + auto-applies; SessionStart hook re-applies a committed posture); manual `/posture-dashboard` in a separate test project.

### Residual risks & resolutions

| Risk / question | Resolution |
|---|---|
| Two near-identical servers drift | **Refactor, don't fork** — one script, env-dependent default root |
| serve-dir vs project-root write-escape | Anchor the traversal guard to `PROJECT_ROOT`; re-fuzz in audit-gates |
| Server in the web container | **Don't** — web rules it out; document as a wrong turn |
| Novice stalls on the dev-grade grid | Presets-first Simple mode; Advanced hidden |
| `python3` absent | Launcher prints the conversational-fallback path |
| Commit YAML only vs YAML+`settings.json` (web) | **Commit both** + the re-apply hook |
| Server has no auth | Keep `127.0.0.1` default (already warns on `0.0.0.0`/forwarded) |
| `/posture-dashboard` vs `/set-posture --ui` | Lean new command for novice discoverability; cross-link |
