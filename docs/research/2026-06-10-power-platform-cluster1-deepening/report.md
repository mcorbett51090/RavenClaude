## The one-paragraph verdict

Microsoft shipped its own agentic Power Platform tooling into this exact space and it has **already subsumed the core of Find 2 and overlaps Find 4** — `microsoft/Dataverse-skills` (Apr 2026) + the official Dataverse MCP now do schema-write-bound-to-solution, and `microsoft/power-platform-skills` ships an official `code-apps` plugin. The two finds with **genuinely uncovered territory** are **Find 1** (Greg.Xrm.Mcp AppMaker — a *UI-authoring* layer: FormXML / FetchXML+LayoutXML views / model-driven-app + **SiteMap** editing that no official MCP tool touches) and **Find 3** (the Power Automate **flow-lifecycle + tenant-governance** plane — DLP / env-admin / ALM — that the data-centric official surface doesn't address). **Find 4 (zyborc Code Apps gotchas) survives subsumption** and is the **clearest fold-in**: author our own skill, crediting it. **The strategic headline:** `microsoft/power-platform-skills` + `microsoft/Dataverse-skills` are Microsoft's *own Claude Code / Copilot skill marketplaces* — a **direct structural analog (and competitor) to RavenClaude's `power-platform` plugin.**

## Per-find

### Find 1 — neronotte/Greg.Xrm.Mcp (+ PACX) → **PARTIALLY-SUBSUMED; narrow genuine moat**
- **Depth — VERIFIED real.** A genuine MCP *framework* (`Greg.Xrm.Mcp.Core`: OAuth, metadata/query, stdio+SSE scaffolding) with the flagship **AppMaker** server doing AI-driven FormXML get/update/validate, view create via **FetchXML+LayoutXML**, model-driven-app + **SiteMap XML** authoring, with an explicit "LLM edits are non-deterministic — back up first" caveat. **PACX** (`Greg.Xrm.Command`) is a genuine pac-CLI extension *platform* (XrmToolBox-style, commands shipped as NuGet DLLs). Sources: [Greg.Xrm.Mcp](https://github.com/neronotte/Greg.Xrm.Mcp), [PACX](https://github.com/neronotte/Greg.Xrm.Command).
- **Maturity:** MCP 19★ / 14 releases (latest ~Jan 2026); PACX 143★ / **91 releases, ~monthly** (latest ~May 2026). Both **solo** (Riccardo Gregori, MVP) — bus-factor 1 is the dominant risk. *Don't lend PACX's 143★ maturity to the 19★ MCP repo.*
- **Subsumption — PARTIALLY-SUBSUMED.** The official Dataverse MCP **tool surface has zero FormXML/view/sitemap tools** (data + record/table CRUD only — [tool-shape blog, 2026-06-08](https://www.microsoft.com/en-us/power-platform/blog/2026/06/08/dataverse-mcp-server-understanding-the-new-tool-shape/)). **BUT** the official `microsoft/Dataverse-skills` `dv-metadata` ships a [`forms-and-views.md`](https://github.com/microsoft/dataverse-skills/blob/main/.github/plugins/dataverse/skills/dv-metadata/references/forms-and-views.md) that **already teaches an agent to GET/PATCH FormXML + POST views (FetchXML+LayoutXML) + PublishXml via the Web API** — so AI-driven form/view editing is *covered* (as a taught pattern, if not a discrete tool). **Genuinely OWNED, narrow:** (a) **SiteMap / app-module navigation editing** (absent from both official tool + skill), (b) **FormXML validation beyond Microsoft's stale XSD** — the author documented MS's published FormXML XSD is frozen at Feb-2022/v9.0.0.2090 and ~9–21% of live forms fail it ([dev.to writeup](https://dev.to/_neronotte/when-dataverse-forms-dont-validate-the-xsd-gap-in-microsofts-standard-definitions-1fe2)), and (c) **the NuGet CLI-plugin model** (MS has PAC CLI but no third-party command-plugin marketplace).
- **Fold-in → thin KNOWLEDGE FILE + watch.** Capture two durable, transferable lessons in `plugins/power-platform/knowledge/dataverse-ui-xml-editing.md`: the **FormXML XSD-drift trap** (validate structurally, not just against the stale XSD; back up the solution; expect non-determinism on LLM edits) and the **PACX CLI-extension pattern** as a reference. Don't build a skill on a moat MS can close from its own roadmap; don't vendor.

### Find 2 — mwhesse/dataverse-mcp → **SUBSUMED + dormant → SKIP**
- **Depth — real but the scout had two errors.** It genuinely does schema-WRITE (tables, 11 column types incl. AutoNumber/Lookup, 1:N + N:N, option sets, security roles/teams/BUs) bound to a solution+publisher prefix persisted in a `.dataverse-mcp` file, + Mermaid-ER from live schema ([repo](https://github.com/mwhesse/dataverse-mcp)). **Correction 1:** it is **NOT npm-published** — install is local-build only; the npm `dataverse-mcp` the scout saw is an unrelated package. **Correction 2:** it is **dormant — last commit 2025-08-15 (~10 months stale).**
- **Subsumption — FULLY / largely SUBSUMED.** The scout's premise ("official MCP is read-only so this owns schema-write") is **false as of 2026-06.** The official Dataverse MCP `create_table`/`update_table` now defines schema *incl. relationships* ([Learn](https://learn.microsoft.com/power-apps/maker/data-platform/data-platform-mcp#list-of-tools)), and **`microsoft/Dataverse-skills`** (official, 138★, MIT, Apr 2026) wraps MCP + PAC CLI + Python SDK behind `dv-metadata` (tables/columns/relationships) + `dv-security` (roles/teams/BUs) + `dv-solution` (lifecycle + auto-add-to-solution + publisher prefix) — i.e. the exact named differentiator. Residue: only Mermaid-ER-from-live-schema and single-server security-authoring are distinct, and both are thin.
- **Fold-in → SKIP the repo; WATCH/EVALUATE `microsoft/Dataverse-skills` instead.** Capture only the **Mermaid-ER-from-live-schema** idea as a backlog line.

### Find 3 — rcb0727/powerautomate-mcp-docs → **OWNED at the MCP layer, but UNAUDITABLE → watch**
- **Depth — UNVERIFIABLE (refuted as auditable).** The GitHub repo is **documentation-only** (5 files: CHANGELOG/CLAUDE/INSTALL/LICENSE/README — **no source**). The actual server is a **closed-source npm binary** (`powerautomate-mcp`, v0.9.3, [npm](https://registry.npmjs.org/powerautomate-mcp)). The **109-tool claim is vendor-asserted in markdown only** — not code-auditable — and **"3 rounds of penetration testing" is an unfalsifiable self-claim** (no report/auditor/scope; closed-source). Real product, but you cannot verify what flow-from-NL or test/debug actually do, or which API they hit.
- **Maturity:** 52★, 0 open issues, ~4 months old, **30 npm versions, pre-1.0, solo.** vs the commercial **Flow Studio MCP** (~25+ tools, freemium $29–99/mo, MS-unaffiliated) — neither is open-source-auditable.
- **Subsumption — PARTIALLY-SUBSUMED (uncovered at MCP, owned at portal).** The official Dataverse MCP is **data-only** — it does **not** touch Power Automate (scout was right *about MCP*). **But** Microsoft owns **NL flow creation + self-healing** at the **Copilot-in-Power-Automate / portal layer** ([copilot-overview](https://learn.microsoft.com/en-us/power-automate/copilot-overview), 2026 wave 1: agent authoring + self-heal). The durable uncovered slice = **agentic Power Automate flow lifecycle + tenant governance (DLP / env-admin / ALM) over MCP, scriptable from a CLI outside the portal.**
- **Fold-in → WATCH + knowledge note (NOT adopt).** The harness flags this governance gap as **the best /forge candidate of the four — but ONLY after independently verifying the tool inventory** (the 109-tool claim is unauditable; don't build on a closed binary's marketing). Capture the gap as a `knowledge/` note: *"agentic PA flow-build/test/governance exists today only via 3rd-party MCPs because the official Dataverse MCP is data-only — watch for MS to close it."*

Sources: [sharepoint.md](https://raw.githubusercontent.com/zyborc/power-apps-code-apps-skill/main/sharepoint.md), [copilot-studio.md](https://raw.githubusercontent.com/zyborc/power-apps-code-apps-skill/main/copilot-studio.md), [author blog](https://zyborc.github.io/blog/2026-03-power-apps-code-apps-claude-skill/). **Code Apps is a real, current, PREVIEW MS framework** ([Learn, dated 2026-05-29](https://learn.microsoft.com/en-us/power-apps/developer/code-apps/overview)) — the `pac code` CLI is being replaced by an npm CLI, so the surface is in flux.

This is the most RC-pattern-shaped, survives subsumption, and is a small, shippable unit.
3. **Knowledge files (thin): Find 1's** FormXML-XSD-drift + PACX-CLI-extension lessons, and **Find 3's** "agentic PA flow/governance lives only in 3rd-party MCPs today" gap note.
4. **Forge candidate (gated): Find 3's** Power Automate flow-lifecycle + governance plane — *the* genuinely-uncovered agentic gap — **but only after independently inventorying the capability** (the 109-tool closed-binary claim is unauditable; build on the verified gap, not the marketing).
