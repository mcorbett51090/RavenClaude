# New-plugin candidates — research & prioritization (2026-06-10)

> Research pass to identify plugins **not yet implemented** in the RavenClaude marketplace, scoped against the ~98 plugins already shipping. Each candidate is justified against a real gap (no existing plugin owns the lane), with an implementation sketch and a feasibility note. The top-priority candidate is **built in this same change** (`plugins/sales-engineering/`); the rest are a prioritized roadmap.

## Method

1. Enumerated the existing roster (`plugins/`, `.claude-plugin/marketplace.json`) — ~98 plugins across the software-delivery chain, the three clouds, app craft, data & AI, the Microsoft stack, and a deep bench of business + industry verticals.
2. Looked for **lanes with no current owner** — cross-checking each candidate against the closest existing plugin so a "new" plugin is a genuine gap, not a re-skin (the house rule against overlap).
3. Prioritized by **(a) breadth of demand** (how many consumers would enable it) × **(b) feasibility** (can it be built to the repo's quality bar — agents with the scenario schema, a cited knowledge bank, decision trees — from durable, evergreen craft rather than volatile facts that rot).

## Prioritization rationale

- **#1 sales-engineering** wins on the demand×feasibility product: it serves essentially every B2B software company, sits in a clean unowned lane next to `sales-revops`/`product-management`, and is built from **durable** craft (MEDDPICC, Command-of-the-Message, Great Demo!, SIG/CAIQ/SOC 2 mapping) — so the knowledge bank won't rot the way a volatile tool-version map would. That makes it the safest first full build.
- **#2 developer-relations**, **#4 grant-writing-management**, **#6 trust-and-safety**, and **#10 revenue-enablement** are the next tier: all high/medium-high demand and **evergreen** (low freshness risk), so they're the lowest-risk follow-on builds.
- **#3 GIS**, **#5 bioinformatics**, **#7 content-marketing-seo**, **#8 quant-trading**, **#9 robotics** are valuable but carry **more volatile facts** (tool versions, algorithm/SERP/SEO mechanics) — each would need dated, re-verify-at-use knowledge per the marketplace freshness discipline, so they cost more to build to the quality bar and slot lower.

## What was built in this change

**`plugins/sales-engineering/` (v0.1.0)** — full plugin to the repo's quality bar (3 agents with the scenario-authoring schema, 5 skills, a 4-doc cited knowledge bank with Mermaid decision trees, 5 templates, a best-practices set, an advisory anti-pattern hook, a scenarios bank, README + CLAUDE.md + CHANGELOG, marketplace + architecture-roster registration).

## Honest scope note (the blocker)

Building **all ten** plugins to this marketplace's quality bar in a single pass is not feasible and would be the wrong call: each plugin here is a multi-file artifact (agents that must carry the full scenario-authoring frontmatter schema, a citation-grounded knowledge bank with decision trees, templates, a hook) gated by CI (`check-frontmatter.py`, `validate-layout`, `check-marketplace-claims`). Shipping ten thin shells would fail those gates and dilute the marketplace. The responsible deliverable is **one excellent, CI-passing plugin now** plus this **prioritized roadmap** for the remaining nine — each can be built as a follow-on PR in priority order.
