# Use sample-activity.json for offline renderer testing, not as a production substitute

**Status:** Absolute rule
**Domain:** Team portfolio / testing
**Applies to:** `team-portfolio`

---

## Why this exists

`templates/sample-activity.json` is a fixed fixture that lets the portfolio renderers be tested without a live GitHub token. It is valuable for verifying the reporting scripts work and for demonstrating the plugin to new users. It is not a fallback for when the live collection fails. A report generated from the sample fixture contains fictional contributors, fictional repos, and counts that do not reflect the team's actual activity — presenting it to a supervisor as if it were live data is a data-quality failure.

## How to apply

The sample fixture's only sanctioned use cases are:

1. **Offline renderer test** — verify `portfolio-report.py` and `portfolio-dashboard.py` produce valid output before any token is configured (portfolio-setup skill Step 4).
2. **Plugin demo** — show a new team the shape of the output before they have configured their own hub.
3. **CI test** — the marketplace's own CI validates the renderers against the sample fixture.

In all other contexts, use the live `portfolio-activity.json` generated by `portfolio-collect.py`.

```shell
# CORRECT — offline test with sample fixture
python3 "$PLUGIN/scripts/portfolio-report.py" \
  --activity "$PLUGIN/templates/sample-activity.json" --out-dir /tmp/reports

# CORRECT — live production use
python3 scripts/portfolio-collect.py --config team-portfolio.json --out portfolio-activity.json
python3 scripts/portfolio-report.py --activity portfolio-activity.json --out-dir reports/
```

**Do:**
- Run the sample-fixture test once during initial setup to confirm the renderers work.
- Label any sample-fixture output clearly: `[SAMPLE DATA — not live]`.

**Don't:**
- Commit sample-fixture output as the scheduled report in the hub repo.
- Use the sample fixture when a token error causes the live collection to fail — fix the token error; do not substitute sample data.

## Edge cases / when the rule does NOT apply

- There are no legitimate exceptions to this rule for production use. If a live collection fails, the correct response is to surface the failure (via the error log banner in the reports), fix the underlying cause, and re-run — not to substitute sample data.

## Provenance

Derived from the plugin's design intent for `templates/sample-activity.json`: it is a test fixture, not a fallback data source. The distinction matters because the output looks identical to a live report; the error is invisible without checking the source.
