# Make consumers idempotent and give them a dead-letter queue

Redelivery is not an exception in streaming — it is guaranteed by at-least-once delivery, rebalances, and replays — so every consumer must produce the same result when it processes the same message twice, via an idempotency key, an upsert, or a dedup store. Pair that with a dead-letter queue for messages that repeatedly fail: a poison message that can't be processed will otherwise block its partition forever or spin in an infinite retry, and the choice is never "lose it silently." Route the unprocessable to a DLQ with its error context, keep the partition moving, and alert on DLQ depth. Idempotence handles the duplicate; the DLQ handles the un-processable. A consumer with neither is a consumer that corrupts data on retry and stalls on a single bad record.
