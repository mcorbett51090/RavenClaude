# Checkpoint stateful operators so a restart isn't a data loss

Any operator that holds state — a windowed aggregation, a stream join, a running counter — must checkpoint that state to durable storage on a regular interval, with offsets committed atomically alongside the state, so a crash or rescale resumes from a consistent point instead of replaying from zero or losing the in-flight window. Without coordinated checkpointing, recovery either re-emits already-processed results (double-counting) or starts cold (losing the partially-accumulated window), and which one you get is luck. Tune the interval against the trade it controls: frequent checkpoints shrink recovery time but add steady overhead; sparse ones are cheaper but cost more reprocessing on failure. State that isn't checkpointed isn't recoverable — and unrecoverable state is an incident waiting for its first restart.
