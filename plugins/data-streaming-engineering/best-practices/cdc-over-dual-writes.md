# Capture database changes with CDC, never dual-writes

When a database change must become an event, read it from the database's commit log (Debezium-style CDC) or publish via a transactional outbox — never have a service write to both its database and the broker in the same request. A dual-write has no shared transaction: the process can crash between the two writes, leaving the database committed but the event lost, or the event sent but the row rolled back, and the divergence is permanent and undetectable. CDC and the outbox both anchor the event to the same commit that changed the state, so the stream is a faithful, ordered replay of what actually happened. The dual-write is the single most common source of silent state/stream divergence; design it out from the start.
