# Data to stdout, diagnostics to stderr

The two standard output streams have distinct jobs, and crossing them breaks composition. **stdout** carries the program's primary data/result — the thing `tool | next` consumes and `tool > file` captures. **stderr** carries everything else: logs, progress bars, prompts, warnings, and errors. Keeping them separate is exactly what lets a user write `tool 2>/dev/null` to see only data, `tool >/dev/null` to see only diagnostics, and `tool | jq` to parse clean output. The classic bug is printing an error or a status line to stdout: now `tool 2>/dev/null` still shows the error, and the next program in the pipe ingests it as data. Rule of thumb — if removing the line would corrupt what a downstream program parses, it belongs on stdout; otherwise it belongs on stderr.
