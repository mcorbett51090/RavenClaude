# Distribute a single binary and ship completions

Install friction is where good CLIs lose adoption. When the language allows it (Go, Rust), ship a **single static binary** per OS/arch — no runtime to install, no dependency hell, just download-and-run — and publish it where users look (Homebrew, Scoop, winget) with checksums kept in sync with the release. For interpreted tools, isolate the install (`pipx` for Python apps, an `npm` package with a `bin` entry for Node) so it doesn't pollute the user's environment. Whatever the form, **package the generated shell completions** (bash/zsh/fish/PowerShell — generated from the parser, never hand-maintained) and a **man page** with the release, so tab-completion and `man tool` work out of the box. If you offer a `curl | sh` installer, it must verify a checksum/signature and be auditable — an unverified one is arbitrary remote code execution. The CI wiring and code-signing of all this route to `devops-cicd`; the design of *what the release produces* is yours.
