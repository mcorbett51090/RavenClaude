# Respect NO_COLOR and detect the TTY

Color, bold text, and progress bars make a CLI pleasant for a human but turn a piped or redirected stream into escape-code soup that downstream parsers choke on and CI logs render as garbage. Gate all styling behind two checks: **`isatty`** on the relevant stream (only colorize when actually attached to a terminal) **and** the cross-tool **`NO_COLOR`** convention (any non-empty value disables color), with **`FORCE_COLOR`** as the explicit override when a user *wants* color through a pipe. Apply the same TTY gate to progress bars, spinners, and interactive prompts — when stdin/stdout isn't a terminal, fall back to plain output or require a flag instead of hanging on a prompt nobody can answer. The default a user never sets should always be the safe one: no codes when not a TTY.
