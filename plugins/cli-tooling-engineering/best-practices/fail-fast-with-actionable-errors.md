# Fail fast with actionable errors

A CLI error message is read by a frustrated human at the worst moment, so it must do three things: say **what** went wrong, **why** (the specific cause, not a stack trace), and **what to do next**. Validate inputs and preconditions **early** — fail before doing half the work and leaving partial state, not after. Write the error to **stderr**, exit a distinct non-zero code, and where you can, name the exact fix ("config file not found at `~/.config/tool/config.toml` — run `tool init` to create one" beats "Error: ENOENT"). Avoid dumping a raw exception or a 40-line stack trace as the primary message — keep that behind `--verbose`/`--debug`. The difference between a tool people tolerate and one they trust is whether its errors point at the solution.
