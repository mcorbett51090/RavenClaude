# Design the command surface first

The command names, subcommand structure, flags, and positional arguments are the part of a CLI you can least afford to change later — once users wire them into scripts, aliases, and CI, a rename is a breaking change. Design the surface before writing the commands: pick a consistent shape (`tool <noun> <verb> [flags]` for multi-feature tools, a flat flag set for single-purpose ones), separate **global** flags from **command-local** ones, make required inputs **positional** and optional toggles **flags with sensible defaults**, and reserve the conventional flags (`-h/--help`, `--version`, `-v/--verbose`, `-q/--quiet`, `--json`). Aim for the common case to need no flags at all. The test: can a new user guess the command from the noun and verb, and can a script author predict the flag from its name?
