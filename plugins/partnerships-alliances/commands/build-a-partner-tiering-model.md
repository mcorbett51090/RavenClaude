# Build a partner tiering model

You are running `/partnerships-alliances:build-a-partner-tiering-model` for `$ARGUMENTS`. Run it the way the team's specialists would — applying the house opinions in [`../CLAUDE.md`](../CLAUDE.md) §3.

## Steps (traverse top-to-bottom; do not skip)
1. Define the obligation side first — certs, sourced pipeline, capacity, references. No obligation, no tier (§3 #2).
2. Match benefits to obligations — every benefit (margin, MDF, leads, support) traces to a commitment.
3. Set qualification thresholds — measurable, time-bound, with promotion/demotion rules.
4. Price the program economics — benefit cost vs cost-to-serve vs sourced revenue; cite external norms with source + date.
5. Set the review cadence — when tiers re-qualify and who owns it.

## Output
A tier model with per-tier obligations, matched benefits, measurable thresholds, program economics, and a re-qualification cadence. See [`../skills/build-a-partner-tiering-model/SKILL.md`](../skills/build-a-partner-tiering-model/SKILL.md) and the tier-design tree in [`../knowledge/partnerships-decision-trees.md`](../knowledge/partnerships-decision-trees.md).

## Guardrails
- Apply the §3 house opinions before any method.
- No partner PII; cite a source + date for every external margin/MDF figure (or mark it).
- End with owner / date / expected movement on each recommendation.
