# Name the objective — latency or throughput, not both at once

Latency and throughput trade off against each other, and a target that doesn't say which it optimizes is two targets fighting. Past the knee, pushing throughput inflates latency (the queue grows); protecting latency caps throughput (you must shed load below the saturation point). You cannot maximize both — so state the objective before the test, not after. A latency-led system holds a p99 ceiling and accepts a lower sustainable RPS; a throughput-led system maximizes RPS and accepts a higher tail under load. When a stakeholder asks for "more throughput *and* lower latency", that is a request to move the knee (optimize the bottleneck), not a single test target — name it as such and route the actual fix. Write the objective into the NFR so the load test, the capacity math, and the sign-off all gate on the same number.
