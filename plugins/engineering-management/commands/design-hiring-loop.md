# Design a hiring loop

You are running `/engineering-management:design-hiring-loop` for `$ARGUMENTS`. Run it the way the team's specialists would — applying the house opinions in [`../CLAUDE.md`](../CLAUDE.md) §3.

## Steps (traverse top-to-bottom; do not skip)
1. Define the bar from the role — what must this person do in 6–12 months → 3–5 competencies (§3 #6).
2. One competency per interview, with a rubric — same bar Tuesday and Friday (§3 #6).
3. Consistent, behavioral questions — same core questions per competency; past behavior with evidence.
4. A debrief that surfaces dissent — written scores before the group talks; dissent is data (§3 #6).
5. Bias + candidate-experience pass — no protected-class/PII in scoring artifacts (§2).

## Output
A structured loop: competencies, per-interview rubric, question bank, debrief protocol. See [`../skills/design-hiring-loop/SKILL.md`](../skills/design-hiring-loop/SKILL.md).

## Guardrails
- Apply the §3 house opinions before any method; resist a single-impression hire/no-hire.
- No protected-class or PII in scoring artifacts; cite a source + date for any external figure (or mark it).
- End with owner / date / expected change on each recommendation.
