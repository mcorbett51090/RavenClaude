# Roll out a new model shadow, then canary, then full

Never swap a production model in one move. Shadow the new version against live traffic first — it scores real requests but its predictions don't affect users — so you can compare its behavior to the incumbent with zero blast radius. Then canary it to a small traffic slice, watch the promotion metric and the operational signals, and only then promote to full. A blind swap is a silent-regression risk: offline metrics never fully predict production behavior, because the live feature distribution, latency profile, and edge cases differ from your evaluation set. The cost is a more elaborate rollout path and the infrastructure to route and compare traffic; the payoff is that a bad model is caught on 1% of users with a one-click rollback instead of discovered on 100% by an angry customer.
