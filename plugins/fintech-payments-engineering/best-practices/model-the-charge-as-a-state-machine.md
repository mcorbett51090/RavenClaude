# Model the charge as an explicit state machine

A payment is not a boolean; it moves through states — created, requires_action, authorized, captured, succeeded, failed, refunded, disputed — and the only safe way to handle it is an explicit state machine with defined transitions, driven by verified webhooks. Ad-hoc boolean flags (`paid`, `failed`) can't represent a partial capture, an async authentication, a late refund, or a chargeback arriving weeks after success, and they let out-of-order webhooks corrupt the record by clobbering a later state with an earlier one. Make transitions explicit and idempotent, reject illegal ones, and let a stale webhook for a state you've already advanced past be a no-op. The state machine is what makes at-least-once, out-of-order webhook delivery survivable; without it, every async or delayed event is a chance to put money in the wrong state.
