# The watchdog is the last line, not the plan

A watchdog recovers from the bug you didn't find; it is not a substitute for finding it. Kick it from a health check that proves real progress — confirm the main task is draining its queue, the sensor is producing fresh samples, the network task is making forward progress — never blindly from a timer ISR that fires regardless of whether the system is actually doing useful work. A watchdog kicked from a free-running timer papers over a hung main loop and turns a diagnosable fault into a silent reboot loop. Use a windowed watchdog where the silicon offers one (it catches a too-fast kick as well as a too-slow one), record the reset cause across the reset so a watchdog reboot is visible in the logs, and treat any watchdog reset in the field as a bug to root-cause, not as the recovery mechanism working as intended.
