# Product Management — best-practice docs

Named, citable rules for the `product-management` plugin's specialists. Each file is **one rule**, grounded in this plugin's knowledge bank and agent opinions — read, applied, and cited as a whole.

These complement, and do not restate, the cross-cutting house opinions in the team constitution ([`../CLAUDE.md`](../CLAUDE.md) §2) or the inherited `ravenclaude-core` protocols. They take one opinion each and make it a standalone, exception-documented rule.

## Index

_22 rules. Each file is one named, citable rule; read and apply it whole._

| Doc | Status | Use when |
|---|---|---|
| [`avoid-the-build-trap.md`](./avoid-the-build-trap.md) | Absolute rule | Success is being measured by features shipped rather than customer or business value — the build trap is an organizational failure mode, not just a PM mistake. |
| [`discovery-is-continuous.md`](./discovery-is-continuous.md) | Absolute rule | Planning or building anything without current, direct customer contact — discovery is a rhythm, not a phase. |
| [`love-the-problem-not-the-solution.md`](./love-the-problem-not-the-solution.md) | Absolute rule | A spec or proposal describes a solution without framing the problem it solves or the outcome it targets. |
| [`north-star-with-input-metrics.md`](./north-star-with-input-metrics.md) | Absolute rule | Choosing or reviewing the product's primary success metric — the North Star must capture value delivered, not activity. |
| [`opportunity-solution-tree.md`](./opportunity-solution-tree.md) | Pattern | Structuring a discovery space — map outcomes to opportunities to solutions before jumping to a specific feature. |
| [`outcomes-over-outputs.md`](./outcomes-over-outputs.md) | Absolute rule | A team or initiative is being evaluated on what was shipped rather than what changed for the customer. |
| [`prioritize-by-evidence.md`](./prioritize-by-evidence.md) | Absolute rule | A roadmap decision needs to be made and the loudest voice in the room is the current decision-making mechanism. |
| [`product-owns-what-why.md`](./product-owns-what-why.md) | Absolute rule | A decision is on the table and it's unclear whether it belongs to product or project-management. |
| [`roadmap-in-bets-with-confidence.md`](./roadmap-in-bets-with-confidence.md) | Pattern | Building or reviewing a product roadmap — each item should be a named bet with a stated confidence level. |
| [`say-no-and-kill-features.md`](./say-no-and-kill-features.md) | Pattern | A feature backlog keeps growing and nothing is ever removed — maintenance + complexity tax is accumulating. |
| [`ship-to-learn.md`](./ship-to-learn.md) | Pattern | Scoping a release — always ship the smallest thing that produces a real learning, not the full feature. |
| [`test-the-riskiest-assumption-first.md`](./test-the-riskiest-assumption-first.md) | Absolute rule | An initiative is about to be built — surface and test the riskiest assumption before committing engineering effort. |
| [`activation-is-the-first-metric-to-fix.md`](./activation-is-the-first-metric-to-fix.md) | Primary diagnostic | Retention is the focus of improvement work — but only run retention analysis on cohorts with healthy activation. |
| [`assumption-map-before-you-build.md`](./assumption-map-before-you-build.md) | Absolute rule | An initiative is ready to enter design — map and rank all assumptions by importance × evidence quality before any spec is written. |
| [`build-buy-partner-decision-is-strategic-not-just-technical.md`](./build-buy-partner-decision-is-strategic-not-just-technical.md) | Pattern | A capability decision arises — evaluate whether owning it gives a durable competitive advantage before defaulting to build. |
| [`interview-for-the-job-not-the-feature.md`](./interview-for-the-job-not-the-feature.md) | Absolute rule | Customer interviews are being structured around feature feedback — restructure them around the job-to-be-done and past behavior. |
| [`prd-names-the-constraint-not-just-the-problem.md`](./prd-names-the-constraint-not-just-the-problem.md) | Pattern | Writing a PRD — include an explicit constraints section so the engineering team can innovate within the right envelope. |
| [`scope-the-mvp-by-the-learning-goal.md`](./scope-the-mvp-by-the-learning-goal.md) | Pattern | Scoping an MVP — define the learning goal and minimum evidence first, then derive the scope, not the other way around. |
| [`segment-before-you-analyze-retention.md`](./segment-before-you-analyze-retention.md) | Primary diagnostic | Running a retention analysis — segment by acquisition source and activation status before drawing any conclusions from the aggregate curve. |
| [`strategy-has-a-target-customer-not-everyone.md`](./strategy-has-a-target-customer-not-everyone.md) | Absolute rule | A product strategy or roadmap review — the strategy must name a specific target customer with a job-to-be-done AND an explicit non-target. |
| [`weekly-customer-touchpoint-is-infrastructure.md`](./weekly-customer-touchpoint-is-infrastructure.md) | Pattern | A team has not spoken to a customer in more than two weeks — set up a standing weekly touchpoint as a recurring program, not an event. |
| [`write-the-success-metric-before-writing-the-spec.md`](./write-the-success-metric-before-writing-the-spec.md) | Absolute rule | Starting any PRD or initiative brief — write the pre-committed success metric as the first section, before the problem statement. |
