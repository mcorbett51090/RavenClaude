# A denylist is only as good as its coverage — prefer allowlists, and enumerate every bypass

Any filter that decides "block this / scan this" is a security control, and its security is exactly its coverage of inputs and forms — a single missed variant is a full bypass, not a partial one. Two failure shapes recur. First, **matching a name instead of the thing**: a rule that denies `| sh`/`| bash` right after a pipe is defeated by a path-qualified or relative invocation (`curl … | /bin/bash`, `| sudo /bin/sh`, `| ./sh`) because the interpreter's *name* no longer sits where the pattern anchored — you must match the interpreter in all its forms (bare, `/usr/bin/`-qualified, `./`-relative, `sudo`-wrapped, `env`-wrapped), not the one spelling you first thought of. Second, **scanning a subset of the surface**: a secret/PII scanner that checks the obvious fields misses a credential pasted into the one field nobody enumerated (a `ghp_…`/`AKIA…` token in an unscanned metadata field goes to disk verbatim) — the scanner must cover *every* field, path, and encoding an attacker or a careless user can reach, not a representative sample. The durable lessons: **prefer an allowlist** (enumerate the small set of good, deny the rest) over a denylist (enumerate the infinite set of bad) whenever the good set is boundable; when you must denylist, **write a test for each bypass form and pin it as a fixture** so a future edit can't silently reopen it; and treat coverage — not cleverness of the pattern — as the metric. A control that is 95% covered is 0% effective against anyone who finds the other 5%.
