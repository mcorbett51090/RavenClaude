# Draft a CFP abstract

You are running `/developer-relations:draft-cfp-abstract`. Write an accept-able,
attendee-led CFP abstract for `$ARGUMENTS` using the `developer-advocate` discipline
and the [`conference-talk-and-cfp`](../skills/conference-talk-and-cfp/SKILL.md) skill.

## Steps

1. **Reframe to the attendee takeaway** — what will they be able to *do* after the talk?
   The product, if present, is the example, not the subject
   ([`../best-practices/cfp-abstract-leads-with-the-attendee-takeaway.md`](../best-practices/cfp-abstract-leads-with-the-attendee-takeaway.md)).
2. **State 2–4 concrete takeaways**, each an action ("build/diagnose/avoid X"), never "understand X."
3. **Match the track and level**; write reviewer-fit notes.
4. **Keep it engineer-to-engineer** — strip marketing speak; this is not demand gen
   ([`../best-practices/devrel-is-not-demand-gen.md`](../best-practices/devrel-is-not-demand-gen.md)).
5. **Right-size the scope** for the slot — one clear arc.
6. **Fill the template** ([`../templates/cfp-abstract.md`](../templates/cfp-abstract.md)) and run the
   first-sentence test (the abstract must not open with "At <Company>…").

## Guardrails

- If the first sentence is about you or the product, rewrite it.
- "Learn about" is not a takeaway — name the action.
- Don't over-scope; reviewers reject talks that promise more than the slot allows.
- A talk is an awareness play, not a fix for a broken getting-started — fix TTFS first.
