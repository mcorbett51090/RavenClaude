# Review community health

You are running `/developer-relations:review-community-health`. Diagnose
`$ARGUMENTS` using the `developer-community-manager` discipline: a community is a
garden, not a megaphone.

## Steps

1. **Measure the health metrics** — answered-question rate, time-to-first-response,
   returning contributors, sentiment. Member/follower counts are context, not health.
2. **Find the unanswered questions** — each is a public broken window.
3. **Design triage + response SLAs** so answered-question rate and TTFR improve.
4. **Capture canonical answers** back into docs/samples (route recurring Qs to
   `devrel-content-engineer`, recurring pain to `developer-advocate`).
5. **Design the contributor ladder** — user → answerer → contributor → ambassador,
   recognition that rewards genuine help (no pay-to-post).
6. **Capture it** in the
   [`community-health-review`](../templates/community-health-review.md) template.

## Guardrails

- Don't report community health as member/follower count.
- Don't let first questions rot — the broken-window effect kills communities.
