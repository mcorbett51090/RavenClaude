# Vanity metrics are never success criteria

**Status:** Absolute rule
**Domain:** DevRel metrics
**Applies to:** `developer-relations`

---

## Why this exists

Followers, impressions, GitHub stars, talk-attendee counts, and registration numbers are easy
to grow and easy to report — which is exactly why they seduce DevRel programs into measuring
motion instead of outcomes. A program optimized for stars will produce stars and no activated
developers. Worse, when leadership eventually asks "what did DevRel actually do for the
business," a dashboard full of reach numbers has no answer, and the program loses its budget.

## How to apply

A metric is legitimate as a **success criterion** only if it maps to a developer-funnel stage:
awareness → activation → habit → advocacy. Vanity inputs aren't forbidden from existence — they
are leading *reach* signals — but they are never the goal.

| Vanity input | The outcome metric it should become |
|--------------|--------------------------------------|
| Followers / impressions | New developers attributed to content, then activated |
| Talk attendees | Quickstart completions in the week after the talk |
| GitHub stars | Weekly active / retained developers |
| Member count | Median first-response time + % questions answered |

**Do:**
- Pick a north-star that's an activation or habit *outcome*.
- Pair one leading + one lagging metric per funnel stage.
- Report reach as a signal, clearly labelled as input, not as the result.

**Don't:**
- Set an OKR on followers, stars, or impressions.
- Report a vanity number as the headline of a DevRel review.

## Edge cases / when the rule does NOT apply

- Reach metrics are legitimately the *leading indicator* at the awareness stage — track them
  there, just don't call them the program's success.
