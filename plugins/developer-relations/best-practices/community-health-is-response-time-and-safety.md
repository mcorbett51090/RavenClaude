# Community health is response time and safety, not headcount

**Status:** Absolute rule
**Domain:** Community operations
**Applies to:** `developer-relations`

---

## Why this exists

It's tempting to report community success as member count, because the number only goes up and
looks impressive. But a 50,000-member server where questions sit unanswered for days, or where
newcomers get a hostile reception, is actively harming the product — every unanswered or
unwelcoming interaction is a developer churning. A 500-member community where questions get a
useful first response within hours and the code of conduct is enforced evenly is healthier by
every measure that affects the funnel.

## How to apply

Report three health metrics, not headcount: **median first-response time**, **% of questions
answered**, and **sampled sentiment**. Set a first-response SLA and measure against it. Enforce
the code of conduct on an explicit ladder (warn → mute → ban), applied consistently and logged,
with reporter safety handled privately first.

**Do:**
- Treat first-response time as the headline community metric.
- Rota moderation and answering so no single person burns out (a single point of failure).
- Build a visible contributor ladder with curated good-first-issues and mentorship.

**Don't:**
- Report raw member/registration count as community success.
- Leave the code of conduct unenforced — an unenforced CoC is worse than none.

## Edge cases / when the rule does NOT apply

- Very early communities legitimately track growth to confirm there's an audience at all — but
  add the response-time SLA from day one, before the volume makes it hard to retrofit.
