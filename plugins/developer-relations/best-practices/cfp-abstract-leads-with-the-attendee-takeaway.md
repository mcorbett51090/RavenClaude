# A CFP abstract leads with the attendee takeaway

**Stance:** a conference-talk proposal opens with what the *attendee* will be able to
do afterward — not the speaker's title, the company, or the product. Lead with the
takeaway or the talk is rejected and, if accepted, ignored.

## Why

Program committees reject product pitches, and attendees tune them out. The reviewer
and the attendee both buy one thing: "what will I be able to do after this session?"
A talk is a DevRel *awareness* play — its leverage comes from teaching something
transferable, the same authenticity-over-reach rule the team applies everywhere
([`../CLAUDE.md`](../CLAUDE.md)). An abstract that opens with "At <Company>, we built…"
answers the wrong question.

## In practice

**The first-sentence test:** if the abstract's first sentence is about you or the
product, rewrite it to be about what the attendee will be able to do.

- Open with the attendee takeaway ("you'll leave able to diagnose and fix cache stampedes").
- State 2–4 concrete takeaways — each an *action* ("build / diagnose / avoid X"), not
  "understand X."
- Match the track and level explicitly; write reviewer-fit notes.
- Use the product as the *example* the lessons came from, not the subject — this is
  not demand gen ([`devrel-is-not-demand-gen.md`](devrel-is-not-demand-gen.md)).
- Right-size the scope to the slot; one clear arc beats five rushed points.

## Edge cases

- A **keynote** or a **vendor's own-conference track** has different norms — a product
  narrative is expected there. This rule governs general CFP submissions to community
  conferences.
- A **lightning talk** compresses to a single takeaway — still lead with it, just one.
