# Showback/chargeback creates accountability.

**Status:** Pattern. **Constitution:** §3 #6, §4.

## Use when
Any finops & cloud cost deliverable where this question is in play — read, applied, and cited whole.

## The rule
Spend a team can see is spend a team manages; showback (visibility) drives most of the behavior change, and chargeback (real budget) drives the rest. Allocation without showback is a report nobody reads.

## Why it matters
This is house opinion §3 #6, distilled into a citable rule. Practitioners act on these deliverables, so a framing error here is not academic — it sends real operating decisions the wrong way. The rule is cheap to apply and expensive to skip.

## How to apply
- Apply this **before** reaching for a method — it sets the framing, not the conclusion.
- Make every number in the deliverable carry a definition, a window, and a baseline (§3 #1).
- Cite a source + date for any external benchmark, or mark it `[unverified — training knowledge]` (§3 #8).
- When this rule and another both apply, route to [`finops-lead`](../agents/finops-lead.md) to sequence them — overlapping signals usually mean two drivers at once.
- Keep billing/account PII out of the deliverable; route professional/legal determinations to the qualified authority (§2).
- Close with a recommendation that has an owner, a date, and an expected metric movement.

## The anti-pattern this prevents
The §4 failure mode: acting as if "showback/chargeback creates accountability." weren't true — the most common way an analysis quietly misleads the practitioner who acts on it. The plugin's advisory hook flags a deliverable that reads as if this rule were ignored.
