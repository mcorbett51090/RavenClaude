# Federate CI and cross-cloud identities, never share a key

When an external system — GitHub Actions, an on-prem job, or an AWS/Azure workload — needs to act on GCP, configure Workload Identity Federation so it exchanges its own native OIDC/SAML token for short-lived GCP credentials, impersonating a scoped service account. The alternative, a downloaded SA key handed to the CI system, is a long-lived secret living in a third party's variable store, rotated rarely and revoked never. Scope the federation's attribute conditions tightly — to the specific repository and branch, or the specific external identity — so only the intended caller can assume the account, and grant that account only the roles the deploy needs. The federated token expires in minutes and carries the caller's verifiable identity; the key file is a forever-valid bearer secret waiting to leak.
