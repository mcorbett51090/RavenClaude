# Match the index to the predicate, not the column

An index helps only when its shape matches how the query filters and sorts. A composite index serves predicates on a prefix of its columns, in order — `(tenant_id, created_at)` answers "this tenant, newest first" but not "all tenants by date," so column order is a decision, not an alphabetisation. A partial index (`WHERE status = 'active'`) is smaller and faster when queries always carry that filter; a covering `INCLUDE` index lets the read skip the heap entirely; an expression index is the only thing that makes a function-wrapped predicate sargable. Indexing the column "because it's queried" without matching the predicate's shape, order, and selectivity builds an index the planner ignores while still paying its write cost.
