# Change enablement balances speed and risk, not bureaucracy

**Status:** Absolute rule. **Constitution:** §2 #2.

## Use when
Any ITSM deliverable where this question is in play — read, applied, and cited whole.

## The rule
The goal is to maximize successful changes, not to slow every change to a crawl. A change process that makes the safe path the slow path gets routed around — making things less safe, not more.

## Why it matters
This is a house opinion distilled into a citable rule. IT teams and users live with these decisions daily; a service-management process that ignores this rule doesn't fail loudly — it erodes trust ticket by ticket, breach by breach. The rule is cheap to apply and expensive to skip.

## How to apply
- Apply this **before** reaching for a practice — it sets the framing, not the conclusion.
- Set the assessment depth and approval level to the change's risk, not to a fixed gate for everything.
- Measure change success rate, not the number of approval gates.
- If people are working around the process, that is the finding — redesign it.
- Cite a source + date for any benchmark, SLA target, or tool capability, or mark it `[unverified — training knowledge]` / `[ESTIMATE]`.
- When this rule and another both apply, route to [`service-management-lead`](../agents/service-management-lead.md) to sequence them.

## The anti-pattern this prevents
A one-size CAB that reviews every change, so teams bypass it for anything urgent and the process protects nothing. The plugin's advisory hook flags a deliverable that reads as if this rule were ignored.
