# Standard changes are pre-authorized, not CAB-bottlenecked

**Status:** Absolute rule. **Constitution:** §2 #3.

## Use when
Any ITSM deliverable where this question is in play — read, applied, and cited whole.

## The rule
Low-risk, repeatable changes get a pre-approved model and ship without a meeting. Reserve the CAB for genuinely novel, higher-risk normal changes.

## Why it matters
This is a house opinion distilled into a citable rule. IT teams and users live with these decisions daily; a service-management process that ignores this rule doesn't fail loudly — it erodes trust ticket by ticket, breach by breach. The rule is cheap to apply and expensive to skip.

## How to apply
- Apply this **before** reaching for a practice — it sets the framing, not the conclusion.
- Identify repeatable low-risk changes and build standard-change models for them.
- Keep the CAB for novel/higher-risk normal changes; emergency changes use the expedited path + retrospective review.
- The lever for a faster change process is moving more changes into standard models, not adding reviewers.
- Cite a source + date for any benchmark, SLA target, or tool capability, or mark it `[unverified — training knowledge]` / `[ESTIMATE]`.
- When this rule and another both apply, route to [`service-management-lead`](../agents/service-management-lead.md) to sequence them.

## The anti-pattern this prevents
Sending a weekly, well-understood, low-risk change to the CAB every time — a meeting that adds delay and zero safety. The plugin's advisory hook flags a deliverable that reads as if this rule were ignored.
