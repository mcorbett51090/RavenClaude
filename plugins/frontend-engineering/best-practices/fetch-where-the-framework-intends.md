# Fetch where the framework intends, and avoid waterfalls

Fetch data at the level the framework is built for — RSC/loaders/route data on the server where the runtime can parallelise and start requests before render, not deep in a leaf component's `useEffect`. Client-component effect fetching creates request waterfalls (the child can't start its fetch until the parent renders, which waits on the parent's fetch) and an extra round trip versus rendering with data already present. Hoist fetches so independent requests fire in parallel, colocate them with the route that needs them, and reach for client-side fetching only for data that genuinely depends on client interaction or can't be known at request time. Fetching from inside the component tree is the default that quietly turns one slow page into a chain of sequential round trips.
