# Power Apps Code Apps Architecture & Configuration Reference

## power.config.json

This file is the metadata configuration for a Code App. It is **generated and managed
by the CLI** (`pac code init`). Application logic should not directly modify this file.

### Schema

```json
{
  "appId": "<guid>",
  "entryPoint": "./dist/index.html",
  "manifest": {
    "solutionName": "<solution-name>",
    "publisherPrefix": "<prefix>"
  }
}
```

### Fields

| Field | Type | Description |
|-------|------|-------------|
| `appId` | string (GUID) | Unique identifier linking local code to the registered app in the environment |
| `entryPoint` | string (path) | Path to the main HTML file (output of Vite build) |
| `manifest.solutionName` | string | Target solution for deployment |
| `manifest.publisherPrefix` | string | Publisher prefix (2-8 chars, alphanumeric, starts with letter, cannot start with "mscrm") |

### Important Rules

- This file is auto-generated by `pac code init`
- Do NOT create it manually
- Do NOT edit `appId` -- it maps to the registered app in Dataverse
- If this file is missing, run `pac code init` again
- If `appId` is invalid, the deployment will fail silently

## Project Directory Structure

After scaffolding with the starter template:

```
my-app/
├── public/                  # Static assets
├── src/                     # Your application code
│   ├── App.tsx              # Root React component
│   ├── main.tsx             # Entry point
│   ├── components/          # Reusable components
│   ├── pages/               # Route pages
│   └── styles/              # CSS / Tailwind
├── generated/               # Auto-generated by pac CLI
│   ├── services/            # Typed service classes per data source
│   │   ├── ContactService.ts
│   │   └── AccountService.ts
│   └── models/              # TypeScript type definitions
│       ├── ContactModel.ts
│       └── AccountModel.ts
├── dist/                    # Build output (Vite)
├── power.config.json        # App metadata (CLI-managed)
├── vite.config.ts           # Vite configuration
├── tsconfig.json            # TypeScript configuration
├── package.json             # Dependencies and scripts
└── index.html               # HTML entry point
```

### Key Directories

| Directory | Purpose | Who Creates It |
|-----------|---------|---------------|
| `src/` | Your application code | You (the developer) |
| `generated/` | Typed services and models | `pac code add-data-source` |
| `dist/` | Production build output | `npm run build` (Vite) |
| `power.config.json` | App registration metadata | `pac code init` |

### Files You Edit
- Everything in `src/`
- `vite.config.ts` (to add plugins)
- `package.json` (to add dependencies)
- `tsconfig.json` (to adjust TypeScript settings)
- `index.html` (to change the HTML shell)
- `tailwind.config.js` (if using Tailwind)

### Files You Do NOT Edit
- `power.config.json` (managed by CLI)
- Anything in `generated/` (managed by CLI, regenerated on data source changes)

## Runtime Architecture

### Three-Layer Model

```
┌──────────────────────────────────────────────┐
│  Layer 1: YOUR CODE                          │
│  React/Vue components, business logic        │
│  Calls SDK APIs and generated services       │
├──────────────────────────────────────────────┤
│  Layer 2: POWER APPS SDK                     │
│  @microsoft/power-apps npm package           │
│  Exposes public APIs, manages generated      │
│  models/services, routes data requests       │
├──────────────────────────────────────────────┤
│  Layer 3: POWER APPS HOST                    │
│  Manages Entra ID authentication             │
│  Controls app loading and initialization     │
│  Presents error messages on load failures    │
│  Handles failed app states                   │
└──────────────────────────────────────────────┘
```

### Authentication Flow (Implicit -- No Code Needed)

1. User navigates to the app URL
2. Power Apps Host validates the Entra ID session
3. If not authenticated, Host redirects to Entra ID login
4. After auth, Host loads the app code
5. App code calls `getContext()` -- user info is already available
6. App code calls generated services -- tokens are injected by the Host
7. Connector consent dialogs appear automatically if needed

**You NEVER write auth code. The Host does everything.**

### Data Flow

1. App code calls a generated service method (e.g., `ContactService.getAll()`)
2. SDK translates the call into a Power Platform connector request
3. SDK passes the request to the Host
4. Host injects the auth token and routes to the connector
5. Connector returns data through the Host
6. SDK deserializes and returns typed data to your code

## Constraints

- App must be an HTML-based Single-Page Application
- Built on TypeScript or JavaScript
- Must integrate with Power Platform CLI and Power Apps SDK
- Must use the `powerApps()` Vite plugin
- The `tsc -b && vite build` build chain is required for `pac code push`
