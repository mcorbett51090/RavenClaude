# Automate a11y and visual regression where they earn their keep

Accessibility and visual correctness are real requirements that manual testing checks inconsistently and forgets under deadline. Automate the parts machines do reliably: an axe-style a11y scan in CI catches the mechanical WCAG failures (missing labels, contrast, ARIA misuse) and belongs in the gate — but it's a floor, not proof of usability, so it complements rather than replaces real assistive-tech testing of critical journeys. Visual regression earns its place on stable, high-traffic surfaces where an unintended pixel change is a real defect; it does *not* belong on churning, frequently-restyled pages, where it generates a snapshot-diff treadmill that trains the team to rubber-stamp updates — which is just flakiness with extra steps. Add each only where the defects it catches outweigh the maintenance it demands.
