# Buy feedback speed with sharding before you cut tests

A suite slow enough that developers avoid running it has already failed, no matter how good its assertions are — feedback latency is a first-class property of a test suite. Before deleting valuable tests to go faster, parallelize: shard the suite across runners by historical timing (not file count, or one slow shard gates the rest), run independent levels concurrently, and cache what's deterministic. This is *enabled* by the discipline that already pays off elsewhere — isolated test data and determinism are the precondition for safe parallelism; order-dependent tests can't be sharded. Target wall-clock, not total CPU-time: ten minutes of tests across ten runners is one minute of waiting. When sharding stops helping, *then* move the genuinely slow long-tail post-merge and triage which tests earn their cost.
