# Contract-test the seams so you don't need full integration

Two services that talk over an API can drift apart silently — the provider renames a field, the consumer breaks in production, and neither team's unit tests noticed. Spinning up both for an end-to-end test is slow and flaky and scales badly. Contract tests are the cheaper, faster seam: the consumer declares the requests it makes and responses it expects, and the provider verifies it still satisfies every consumer's contract in *its own* pipeline. The provider learns it broke a consumer before merge, without the consumer's environment. The discipline is to test the *contract* (the shape and semantics both sides agreed) not the implementation behind it. (Pact/consumer-driven mechanics live with `api-engineering`; the strategic call to put a contract test at a service boundary is ours.)
