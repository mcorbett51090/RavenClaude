# Declare required_providers with pessimistic version constraints

**Status:** Absolute rule
**Domain:** IaC / version pinning
**Applies to:** `terraform-iac`

---

## Why this exists

A Terraform configuration without `required_providers` version constraints makes `terraform init` non-deterministic: running `init` today versus tomorrow may pull a different provider version with changed defaults, removed arguments, or breaking resource behavior. The house opinion #5 is explicit: "Pin providers and modules. Floating versions make terraform init non-deterministic." The `.terraform.lock.hcl` file locks the selected version for a given machine, but lock files are not a substitute for declared constraints — constraints define the allowed range; lock files record the specific version chosen.

## How to apply

```hcl
# terraform.tf or versions.tf — declare at the root and in each module
terraform {
  required_version = "~> 1.8"   # patch-upgradeable; never major/minor without review

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # allow 5.x patch updates; not 6.x
    }
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 3.90"  # allow 3.x patch; not 4.x (breaking changes in major)
    }
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
    kubernetes = {
      source  = "hashicorp/kubernetes"
      version = "~> 2.27"
    }
    random = {
      source  = "hashicorp/random"
      version = "~> 3.6"
    }
  }
}
```

```bash
# Lock file: generated by terraform init; ALWAYS commit it
# .terraform.lock.hcl — commit, do NOT gitignore
git add .terraform.lock.hcl
```

**Constraint syntax reference:**

| Constraint | Meaning | Use for |
|---|---|---|
| `~> 5.0` | >= 5.0, < 6.0 | Providers — allows patches, blocks major |
| `~> 5.47` | >= 5.47, < 6.0 | Tighter — allows only patches from this minor |
| `>= 5.0, < 6.0` | Explicit range | When ~> doesn't express the intent clearly |
| `= 5.47.0` | Exact pin | Rarely: when a known-good version must not drift |

**Do:**
- Use `~> MAJOR.MINOR` (e.g., `~> 5.0`) as the standard constraint — allows patch updates, blocks minor/major.
- Commit `.terraform.lock.hcl` — it records the exact hash of the selected provider version.
- Update constraints and lock files intentionally (via `terraform init -upgrade`) during a dedicated upgrade PR, not as a side effect of another change.
- Add `required_version` for Terraform itself alongside `required_providers`.

**Don't:**
- Leave `required_providers` empty or absent — you get whatever is latest at init time.
- Use `>= 5.0` without an upper bound — it allows major version upgrades silently.
- Gitignore `.terraform.lock.hcl` — it is a reproducibility artifact, not a generated build artifact.
- Run `terraform init -upgrade` in a pipeline without reviewing what changed.

## Edge cases / when the rule does NOT apply

- **Published module libraries** (not root modules): modules should declare `required_providers` with wider ranges (e.g., `>= 5.0`) to avoid conflicting with the root module's constraint and let the root pin the specific version.
