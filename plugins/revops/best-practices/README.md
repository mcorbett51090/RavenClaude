# RevOps best-practices

Atomic, enforceable rules the revops agents apply. Each file is one rule with a short rationale; the agents cite them by filename. Canonical decision logic lives in [`../knowledge/revops-decision-trees.md`](../knowledge/revops-decision-trees.md); these rules are the always-on priors.

| Rule | Gist |
|---|---|
| one-funnel-one-definition | One definition of each stage across marketing/sales/CS, instrumented once |
| funnel-is-a-bowtie | Model acquisition and retention/expansion as one connected motion |
| stage-is-exit-criteria-not-vibes | Every stage has an objective buyer-action exit criterion |
| forecast-is-a-methodology | Name the method, its inputs, and its known bias |
| inspect-pipeline-before-the-math | Coverage/win-rate/velocity only on inspected pipeline |
| coverage-is-derived-from-win-rate | Required coverage = gap ÷ win-rate, never a folk 3x |
| routing-and-scoring-are-slas | A defined owner, a speed-to-lead clock, a validated score |
| attribution-is-a-lens-not-truth | Name the model and its distortion; no single model owns budget |
| comp-drives-behavior | Name what a comp/quota/territory design accidentally rewards before it ships |
| quota-is-bottoms-up-from-capacity | Reconcile the board number against ramped-rep capacity, never the reverse |
| data-quality-is-the-substrate | Enforce required fields + dedupe at entry, not in a quarterly scrub |
| the-build-belongs-to-the-system-layer | Specify the contract; hand the CRM/warehouse/dashboard build to the system layer |
