# Apple Sign In Secrets Expire in 180 Days and Must Be Actively Rotated

**Status:** Absolute rule
**Domain:** Auth & Identity — Apple SSO
**Applies to:** `auth-identity`

---

## Why this exists

Apple Sign In's client secret is not a static credential — it is a **JWT signed with an ES256 private key**, and it has a hard maximum lifetime of **180 days**. When the secret expires, every Apple Sign In attempt in the app fails with a silent `invalid_client` error until a new secret is generated and deployed. Unlike Google's OAuth client secret (which does not expire unless rotated manually), Apple's expiring secret is a **mandatory operational event** that will recur every 180 days unless an automated rotation process is in place. The `social-and-passwordless-providers-2026.md` knowledge doc calls this out as the primary Apple gotcha.

## How to apply

**Secret generation formula (Node.js example):**

```javascript
import jwt from 'jsonwebtoken';
import fs from 'fs';

const clientSecret = jwt.sign({}, fs.readFileSync('AuthKey_XXXXXXXXXX.p8'), {
  algorithm: 'ES256',
  expiresIn: '180d',       // max 180 days
  audience: 'https://appleid.apple.com',
  issuer: APPLE_TEAM_ID,   // 10-char team ID from developer.apple.com
  subject: APPLE_CLIENT_ID, // the Services ID (web) or bundle ID (native)
  keyid: APPLE_KEY_ID,     // the key ID of your Sign In with Apple key
});
```

**Rotation checklist:**

- [ ] A calendar reminder is set for **day 150** (30 days before expiry) to regenerate the secret.
- [ ] The new secret is deployed to production before day 180.
- [ ] The secret is stored in a secret manager (Vault, AWS Secrets Manager, Supabase env vars) — **never in source control**.
- [ ] Supabase (or the auth provider) is updated with the new secret before the old one expires.
- [ ] A rotation runbook exists documenting the steps, the credential locations, and the responsible owner.

**Automated rotation (recommended):**

```yaml
# GitHub Actions: scheduled workflow to alert 30 days before expiry
on:
  schedule:
    - cron: '0 9 * * 1'   # every Monday at 09:00
jobs:
  check-apple-secret-expiry:
    runs-on: ubuntu-latest
    steps:
      - name: Check expiry
        run: python scripts/check-apple-secret-expiry.py
        # Decodes the current secret JWT, checks exp, posts a Slack alert if < 30 days remain
```

**Do:**
- Set `expiresIn` to **no more than 180 days** — Apple rejects longer values.
- Treat the `.p8` private key file with the same security as a root CA key: rotate it if compromised, never commit it.
- Test the rotation in staging before applying it to production.

**Don't:**
- Set a calendar reminder and then miss it — automate the alert.
- Re-use the same `expiresIn` timestamp across deployments — each generated secret must have a fresh `iat` and `exp`.
- Confuse the Services-ID (`aud` for web) with the App bundle ID (`aud` for native) — using the wrong one gives `invalid_client` on the first call.

## Edge cases / when the rule does NOT apply

- **Native iOS apps using Sign in with Apple via the AuthenticationServices framework**: the client-secret JWT is generated by Apple's SDK on-device for the native flow; this server-side secret generation applies only to the web/REST callback flow.
