# Split before you touch the data — design the validation scheme first

Decide the train/validation/test split *before* engineering a single feature or fitting a single transform, and make the scheme match how the model will be used. A random split leaks when rows share a group (the same customer in train and test) or a timeline (predicting the past from the future) — use `GroupKFold` when rows cluster and `TimeSeriesSplit` (train on the past, score the future) when order matters. Hold a final test set out of sight until the very end; every peek to tune is a touch, and a test set touched more than once stops being a test set. Check the split's ratios and class balance before trusting any score: a 60/20/20 that puts all the rare class in one fold is a broken evaluation, not a small one.
